# 文本→视频帧编码方案（单帧直传）

## 背景
- 在仅允许 RTMP 的内网环境，将大模型流式输出通过“视频帧承载数据”的方式穿透网络限制。
- 当前默认策略为“明文单帧直传”：每个 LLM 的流式 chunk 会被编码到一张固定分辨率的图像帧里，经 H.264 编码后走 RTMP 推拉。

## 编码参数
- 分辨率：由配置 `video.width/video.height` 决定
- 块大小：由代码常量 `BLOCK_SIZE` 决定（一个块编码 1bit）
- 单帧最大可承载字节数（不含帧头）：`(WIDTH/BLOCK_SIZE * HEIGHT/BLOCK_SIZE) / 8 - 12`
- 帧头（12B）：Magic(4) + Seq(4) + Len(4)

## 单帧负载
- 当前默认将“完整文本”直接作为 payload 写入帧中（不做 JSON 包裹、不做分片字段）。
- 若 payload 超过单帧容量，会被截断到最大可承载字节数；建议保证单次 payload 不超过单帧上限。

## 编码流程
1. 文本转 UTF-8 字节数组；写入帧头 Magic/Seq/Len。
2. 生成图像帧：将帧头+负载映射为像素块（白=1，黑=0），其余区域保持黑色。
3. 推送到 RTMP；可按需设置冗余发送提升接收可靠性。

## 解码流程
1. 拉流端逐帧解析：读取帧头校验 Magic，按 Len 取出负载字节。
2. 将负载转换为字符串，作为“本帧携带的文本”透传给前端。

## 容量估算
以 BLOCK_SIZE=16 举例（仅用于容量估算）：
- 640×480：blocks=40×30=1,200 → 字节=150 → 有效≈138B
- 1280×960：blocks=80×60=4,800 → 字节=600 → 有效≈588B
- 3840×2160(4K)：blocks=240×135=32,400 → 字节=4,050 → 有效≈4.0KB
- 7680×4320(8K)：blocks=480×270=129,600 → 字节=16,200 → 有效≈16KB

## 常见限制
- OpenH264 编码器存在分辨率面积上限（width×height 受限），8K 可能初始化失败。
- 推拉分辨率必须一致，否则采样网格对不齐，解码会长期无输出。
- 块越小容量越大，但抗压缩更差，帧头更易误码（表现为大量帧被丢弃，解码频率降低）。
